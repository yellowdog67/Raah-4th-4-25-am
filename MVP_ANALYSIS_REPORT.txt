1. EXECUTIVE SUMMARY
================================================================================

RAAH is a voice-first AI travel companion for iOS. The user walks, taps an
animated orb, and has a real-time voice conversation with an AI that knows
nearby landmarks, restaurants, weather, and safety conditions.

CODEBASE:
  - 48 Swift files, ~15,000 lines
  - iOS 17+, SwiftUI, @Observable pattern
  - Zero third-party dependencies (pure Apple frameworks)
  - Builds clean on Xcode with iOS 26.2 SDK

ROADMAP COMPLETION:

  - Sprint 2 "Make It Smart" (7 items):      7/7 DONE
  - Sprint 3 "Safe & Monetizable" (6 items): 6/6 DONE
  - Bonus features not on roadmap:           6 additional
  - TOTAL: 24/24 roadmap items + 6 extras = 30 features

PRODUCTION READINESS:

  - 5 features have medium-severity issues
  - Estimated fix time for all blockers: ~4 hours

2. ARCHITECTURE OVERVIEW
================================================================================

DATA FLOW:
  User speaks → LocationManager detects 100m movement →
  ContextPipeline fetches in parallel (Overpass POIs + Wikipedia + Weather +
  Safety + Affiliate) → SpatialCache returns cached, fetches misses →
  ContextValidator self-heals gaps → SpatialContext built →
  System prompt injected into OpenAI Realtime API → Voice response streamed →
  Interaction stored in ShortTermMemory → At session end, LongTermMemory
  extracts preferences via GPT-4o-mini

KEY DESIGN DECISIONS:
  - @Observable (iOS 17+) instead of ObservableObject
  - AppState is single source of truth, owns all services
  - JSON parsing uses JSONSerialization for API responses, Codable for persistence
  - Combine only for location publisher; everything else is async/await
  - Cache-first strategy: POIs 7d, Wikipedia 30d, Weather 1h, Geocode 7d
  - Graceful degradation: every feature works without optional API keys

REQUIRED API KEY:
  - OpenAI (stored in UserDefaults via Settings)

OPTIONAL API KEYS (features degrade gracefully without):
  - Supabase (cloud memory sync)
  - Google Places (enhanced food POIs)
  - Mappls (India-specific features)
  - GetYourGuide (skip-the-line tickets)
  - Brave Search (web knowledge)
  - GeoSure (safety scoring)

FEATURE 1: Real-Time Voice Conversation
--------------------------------------------------------------------------------
WHAT: Full-duplex WebSocket connection to OpenAI Realtime API. User speaks,
      AI responds with voice. Bidirectional audio streaming.

HOW:  OpenAIRealtimeService.swift manages WebSocket to
      wss://api.openai.com/v1/realtime. Audio captured at hardware rate,
      converted to 24kHz PCM16 mono via AVAudioConverter. Playback resampled
      from 24kHz to hardware rate (usually 48kHz). Server-side VAD with
      threshold 0.5, 300ms prefix padding, 500ms silence duration.

WHY:  This IS the product. Voice-first means the conversation must feel
      instantaneous and natural.

SCALE: Handles AirPods connect/disconnect, speaker switching, audio
       interruptions (phone calls). Echo suppression via suppressMicInput
       flag prevents feedback loops.

IMPLEMENTATION: 10/10
RELIABILITY:    8/10
ISSUES:
  - Model ID is "gpt-4o-mini-realtime-preview-2024-12-17" — 15 months old,
    deprecation risk. Should update to latest stable model.
  - Voice hardcoded to "alloy" — no user choice.
  - No guard against calling startVoiceSession() twice simultaneously.

--------------------------------------------------------------------------------
FEATURE 2: Spatial Context Pipeline
--------------------------------------------------------------------------------
WHAT: On significant movement (>100m), fetches nearby POIs, weather, safety,
      affiliate offers, and Wikipedia enrichment in parallel. Builds a rich
      context payload injected into the AI's system prompt.

HOW:  ContextPipeline.swift binds to LocationManager's significantMovement-
      Publisher (debounced 2s). Uses async let for parallel fetching.
      SpatialCache returns cached data first, only fetches misses. Top 5
      POIs enriched with Wikipedia summaries via TaskGroup. ContextValidator
      auto-repairs incomplete data (missing timezone, location name, etc.).

WHY:  Makes the AI spatially aware. Without this, RAAH is just a voice
      chatbot. With this, it knows what's around you.

SCALE: Handles: no internet (uses cache), slow APIs (parallel + timeout),
       missing data (auto-repair), poor GPS (filters stale/inaccurate).
       Cache hit rates tracked for optimization.

IMPLEMENTATION: 9/10
RELIABILITY:    8/10
ISSUES:
  - CRITICAL: Demo POI fallback. If Overpass returns <4 POIs AND user is
    near Goa (lat 15.2-15.6, lon 73.7-74.0), 6 hardcoded fake restaurants
    are injected (Cafe Bodega, Burger Factory, Martin's Corner, etc.). These
    will appear as real POIs during a demo in Goa. Outside Goa, this fallback
    doesn't trigger, so POI count may be low in sparse areas.
  - No circuit breaker on individual API failures.

FEATURE 3: Proactive Narration
--------------------------------------------------------------------------------
WHAT: AI spontaneously mentions interesting places as you walk past them,
      weighted by your preferences. "Notice that building ahead? That's the
      Basilica of Bom Jesus, a UNESCO World Heritage Site from 1605..."

HOW:  When ContextPipeline refreshes, AppState.checkForProactiveNarration()
      filters nearby POIs: must be notable type (heritage, museum, monument,
      architectural, religious, naturalFeature), within 80m, not already
      mentioned. Candidates scored: preference_match * 0.6 + proximity * 0.4.
      Best candidate's name + Wikipedia summary sent to AI as a text message.
      Rate limited: max 1 per 120 seconds.

WHY:  This is the differentiator. Doc 1's thesis: "proactively weaving a
      narrative based on movement." Without proactive narration, RAAH is
      reactive. With it, RAAH is a companion.

SCALE: Handles: no nearby POIs (no trigger), all POIs already mentioned
       (no trigger), user mid-conversation (waits for idle/paused state),
       rapid movement (120s cooldown prevents spam).

IMPLEMENTATION: 9/10
RELIABILITY:    7/10
ISSUES:
  - Can interrupt natural conversation flow. If AI just finished speaking
    and new context arrives, narration fires immediately. Should queue
    narrations and wait for 3+ seconds of idle.
  - voiceState guard allows .paused, which means narration triggers even
    when user has been idle for 10+ seconds. This is intentional but may
    feel jarring (mic suddenly reactivates to announce something).

FEATURE 4: Turn-by-Turn Walking Navigation
--------------------------------------------------------------------------------
WHAT: "Take me to India Gate" → AI gives step-by-step walking directions.
      Full-screen map with blue polyline, turn markers, camera follow, live
      step progression.

HOW:  DirectionsService.swift tries OSRM (free, global, pedestrian-
      optimized) first, falls back to MKDirections walking, then transit.
      Returns structured NavigationStep objects with instruction, distance,
      coordinate. AppState.startNavigation() sets up state, pushes system
      prompt telling AI to give only current step. checkNavigationProgress()
      runs every 3s via location publisher, uses dual-condition advancement:
      user must have walked >= max(50% of step distance, 25m) FROM where
      step was announced AND be within 25m OF the waypoint.

      NavigationMapView.swift shows: MapPolyline (blue, 5pt), UserAnnotation,
      turn markers (green=completed, blue=current, white=upcoming), red
      destination pin. 3D camera follows user heading at 500m distance, 45°
      pitch. Bottom card shows current instruction with direction icon,
      distance in meters, step counter.

WHY:  Practical utility. "How do I get there?" is one of the most common
      travel questions. Having it built-in with progressive voice guidance
      means users never leave the app.

SCALE: Handles: GPS jitter (dual condition prevents false advances), fast
       walking (distance threshold scales with step length), API failure
       (retries once, then falls back to distance-only result with explicit
       instruction to AI not to hallucinate directions).

IMPLEMENTATION: 9/10
RELIABILITY:    7/10
ISSUES:
  - No off-route detection. If user walks the wrong way, navigation
    silently waits forever. No "you seem to have gone off course" warning.
  - No re-routing capability.
  - Bottom card doesn't show ETA or remaining distance.
  - Turn markers are small (14px) — hard to see at realistic zoom.

FEATURE 5: Temporal Awareness
--------------------------------------------------------------------------------
WHAT: AI never recommends closed places, doesn't suggest food right after
      eating, adapts suggestions to time of day.

HOW:  OpeningHoursChecker (Models.swift) parses OSM opening_hours format
      (supports Mo-Fr ranges, comma days, time ranges, 24/7, lunch breaks,
      overnight, wrap-around). Returns: OPEN NOW, CLOSED, CLOSES SOON,
      CLOSED TODAY. ContextPipeline.buildSystemPrompt() injects:
      - CLOSED places: never recommend, suggest open alternative
      - CLOSES SOON: warn user
      - 90-minute food spam threshold
      - 45-minute sightseeing/shopping repeat threshold
      - Time-of-day rules (7 time bands from pre-7AM to post-10PM)
      - Activity recency detection scans last 3 hours of conversations
        for food/sightseeing/shopping keywords with timestamps

WHY:  Recommending a closed restaurant destroys credibility. Time awareness
      makes RAAH feel genuinely intelligent, not just a Wikipedia reader.

SCALE: Handles: missing opening hours (shows raw string), Sunday closures,
       overnight hours, multiple time ranges per day.

IMPLEMENTATION: 9/10
RELIABILITY:    8/10
ISSUES:
  - Opening hours parser handles most OSM formats but may miss unusual
    patterns (e.g., "PH off" for public holidays, seasonal hours).
  - Activity recency uses keyword matching, not semantic understanding.
    "I don't want food" would still trigger the food-recently-discussed flag.

FEATURE 6: Safety System (Heuristic + GeoSure)
--------------------------------------------------------------------------------
WHAT: Evaluates location safety. Shows safety overlay with alerts, action
      buttons (Walk Me Home, SOS, Share Location).

HOW:  SafetyScoreService.swift tries GeoSure API first (if configured),
      falls back to heuristic scoring: nighttime (21:00-06:00) = moderate,
      thunderstorm weather = escalated. Safety levels: safe, moderate,
      caution, danger. SafetyOverlaySheet shows level with icon, description,
      and action buttons.

WHY:  54.6% of solo travelers are female, 70% cite safety as #1 concern.
      Safety is RAAH's brand differentiator.

SCALE: Works without GeoSure API (heuristic fallback). Weather-based
       alerts pull from Open-Meteo (free, no key).

IMPLEMENTATION: 8/10
RELIABILITY:    7/10
ISSUES:
  - CRITICAL: Safety alert condition is inverted. check_safety_score tool
    handler shows alert only when voice is DISCONNECTED:
    `if report.level < .safe && !realtimeService.isConnected`
    During active voice session (when user needs it most), alert suppressed.
  - SafetyOverlaySheet has no loading state while evaluating. If API is
    slow, sheet shows stale data.
  - Heuristic night window (21:00-06:00) is hardcoded, not sunset-based.

FEATURE 7: Walk Me Home Mode
--------------------------------------------------------------------------------
WHAT: Safety companion. AI stays active, checks in every 3 minutes. If user
      doesn't respond to 2 consecutive check-ins (6+ min silence), auto-sends
      emergency SMS with location.

HOW:  AppState.activateWalkMeHome() sets state, sends SMS to emergency
      contact with Apple Maps link, starts/ensures voice session, updates
      system prompt to "watchful companion" tone, starts 3-minute timer.
      performCheckIn() tracks time since last user response. If 2+ missed,
      calls sendEmergencySMS(). Visual indicator: green shield on HomeView.

WHY:  Nobody else has this. It's the demo moment that makes investors
      lean forward.

SCALE: Handles: no voice session (starts one), user responds (resets
       counter), user says "I'm home" (AI calls deactivate tool).

IMPLEMENTATION: 9/10
RELIABILITY:    7/10
ISSUES:
  - SMS open via UIApplication.shared.open() doesn't confirm user actually
    sent it. System assumes location was shared even if dialog dismissed.
  - No rate limiting on emergency SMS. GPS jitter could theoretically
    trigger multiple alerts.
  - Check-in response detection relies on transcript update callback.
    If user speaks but transcript fails to process, counter won't reset.

FEATURE 8: Quick SOS
--------------------------------------------------------------------------------
WHAT: Triple-tap the orb → 3-second countdown → emergency SMS to saved

HOW:  AppState.triggerSOS() validates emergency contact exists, starts

WHY:  Table stakes for safety-focused travel app. Under-2-second emergency

SCALE: Handles: no emergency contact (button hidden, SOS guard returns),

IMPLEMENTATION: 9/10
RELIABILITY:    8/10
ISSUES:
  - If emergency contact phone is empty, triple-tap does nothing with
    no feedback. User doesn't know why SOS didn't work.

FEATURE 11: Weather-Aware Suggestions
--------------------------------------------------------------------------------
WHAT: AI adapts recommendations based on current weather, temperature,
      visibility, and sunset timing.

HOW:  WeatherService.swift fetches from Open-Meteo (free, no API key):
      current temp, humidity, wind, visibility, weather code, sunrise/sunset,

      - Raining: suggest indoor, warn against viewpoints
      - Low visibility: no sunset/sunrise viewpoints

      - Sunset within 60 min + near viewpoint: proactively mention
      - Hot/cold food: hot soup cold days, cold drinks hot days

WHY:  Prevents embarrassing suggestions ("go watch the sunset" in a
      thunderstorm). Essentially free to implement.

SCALE: 7-day forecast enables forward planning ("Will it rain tomorrow?").
       WMO weather codes mapped to 12 human-readable categories.

IMPLEMENTATION: 9/10
RELIABILITY:    9/10
ISSUES:
  - None significant. Open-Meteo is reliable and free.

FEATURE 13: Tiered Caching Layer
--------------------------------------------------------------------------------
WHAT: Local geohash-indexed cache reduces API calls by 60-80%.

HOW:  SpatialCache.swift implements tiered caching in app's Caches directory:
      - POIs: 7-day TTL, indexed by geohash precision 6 (~1.2km cells)
      - Wikipedia summaries: 30-day TTL, indexed by POI ID
      - Weather: 1-hour TTL, indexed by rounded coordinate
      - Geocode: 7-day TTL, indexed by rounded coordinate
      Thread-safe via DispatchQueue. Atomic writes. Tracks hit/miss ratio.

WHY:  Walking back through an area visited 10 minutes ago shouldn't
      re-fetch everything. Caching makes RAAH fast and cheap to operate.

SCALE: Handles: cache miss (fetches fresh), cache expiry (TTL check),
       concurrent access (dispatch queue), disk persistence (JSON files).

IMPLEMENTATION: 10/10
RELIABILITY:    9/10
ISSUES:
  - No cache size limit. Over weeks of use, cache files could grow large.
    No eviction strategy beyond TTL expiry.

FEATURE 14: Web Search for Local Knowledge

WHAT: "Where's the best samosa near me?" → AI searches Reddit, TripAdvisor,

HOW:  WebSearchService.swift wraps Brave Search API (free tier: 2000

SCALE: Handles: API not configured (returns helpful message), no results

IMPLEMENTATION: 9/10
RELIABILITY:    8/10
ISSUES:
  - Requires Brave Search API key to function. Without it, returns
    "Web search is not configured" message.
  - Site filters hardcoded to 3 sites. Could miss valuable local blogs
    or Yelp results.

FEATURE 15: Post-Activity Feedback
--------------------------------------------------------------------------------
WHAT: After visiting a POI for 10+ minutes and leaving, AI asks "So how
      was [place]? Worth the visit?" Response stored as high-confidence
      preference.

HOW:  AppState.trackNearbyDwell() creates DwellEntry for notable POIs
      within 150m. checkForFeedbackOpportunity() checks: dwell time >= 10
      minutes AND user now >300m away. If triggered, sends feedback question
      to AI. Response parsed and stored as 0.9 confidence UserPreference.
      Rate limited: max 1 feedback request per 15 minutes.

WHY:  Explicit feedback gives 0.9+ confidence vs 0.5-0.7 from inferred
      preferences. Builds high-quality taste profiles.

SCALE: Handles: user never visits (no trigger), quick visits <10 min
       (no trigger), multiple POIs (tracks each separately), rapid
       departures (300m threshold prevents premature triggers).

IMPLEMENTATION: 8/10
RELIABILITY:    7/10
ISSUES:
  - Dwell tracker grows unbounded during a session. No cleanup for POIs
    the user walked past but never visited (stays <10 min).
  - Feedback parsing relies on AI interpretation, not structured response.

FEATURE 16: Inference Cost Throttling
--------------------------------------------------------------------------------
WHAT: Pauses audio streaming when user is silent for 10+ seconds. Saves
      60-70% of API costs during idle walking.

HOW:  OpenAIRealtimeService starts idle timer on every response.done and
      response.cancelled event. After 10 seconds of no speech detected,
      pauses audio capture and sets voiceState to .paused. Resumes on:
      user taps mic button, proactive narration triggers, or Walk Me Home
      check-in fires. Also clears audio buffer on pause.

WHY:  OpenAI Realtime API costs ~$9/hour/user. Pausing during silence
      makes the unit economics viable.

SCALE: Handles: user resumes speaking (speech_started cancels timer),
       proactive narration (resumes capture before sending), system
       events (check-ins resume capture).

IMPLEMENTATION: 9/10
RELIABILITY:    9/10
ISSUES:
  - None significant. Timer-based approach is simple and reliable.

SCALE: Three accuracy modes properly managed across all app states.

IMPLEMENTATION: 9/10
RELIABILITY:    9/10
ISSUES:
  - None significant.

BONUS FEATURE 25: Profile Isolation
--------------------------------------------------------------------------------
WHAT: Per-user data isolation. Each user gets a UUID, all storage keys
      prefixed with first 8 chars of UUID. One user's preferences never
      leak to another.

HOW:  AppState.resolveProfileId() creates or restores active profile.
      All UserDefaults keys use profileKey() helper: "raah_{8chars}_{key}".
      ShortTermMemory and LongTermMemory accept userId in init for scoped
      storage keys. Migration: on first run after update, copies all flat
      "raah_*" keys to prefixed keys automatically.

SCALE: Clean architecture ready for multi-user profile switching (future).

IMPLEMENTATION: 9/10
RELIABILITY:    9/10
ISSUES:
  - No profile switching UI yet (single profile per device for MVP).
  - UsageTracker not yet profile-scoped (usage is device-wide).

SCALE: Handles gracefully if repair fails (marks as unfixable, continues).

7. WHAT STOPS A PERFECT DEMO
================================================================================

If you're demoing RAAH to investors, these are the things that could go wrong,
ranked by likelihood and impact:

HIGH LIKELIHOOD, HIGH IMPACT:

  A. "Why can everyone use it for free?"
     The isProUser bypass means your monetization pitch has no proof.
     When you say "30 min/day free, then paywall" — it doesn't actually
     happen. Fix: 5-minute code change.

  B. "Those restaurant recommendations seem suspicious"
     If you demo in Goa and Overpass is slow, fake Cafe Bodega appears.
     If you demo outside Goa, no fallback and sparse areas show few POIs.
     Fix: 10-minute code change (simulator-only gate).

  C. Voice drops during demo
     WiFi→cellular handoff, brief dead zone, or network congestion.
     Reconnect logic works but takes 1-4 seconds. Orb shows "Reconnecting"
     in amber. Not a bug, but looks bad.
     Mitigation: Demo on stable WiFi.

  D. "What if I walk the wrong way?"
     If you show navigation and deliberately go off-route, nothing happens.
     The app silently waits. Investor asks "shouldn't it tell me I'm going
     the wrong way?" and you have no answer.
     Fix: 1 hour.

MEDIUM LIKELIHOOD, HIGH IMPACT:

  E. Safety overlay shows stale data
     If GeoSure API is slow (>5 seconds), the safety sheet hangs with
     old data. User sees wrong safety level.

  F. SOS with no emergency contact
     If you forgot to set emergency contact in Settings, triple-tap SOS
     does nothing with zero feedback. Embarrassing during safety demo.

  G. OpenAI deprecates model
     The preview model is 15 months old. If deprecated, voice breaks
     completely. No fallback. Low probability for a specific demo day,
     but nonzero.

LOW LIKELIHOOD, MEDIUM IMPACT:

  H. "What do I say to it?"
     User starts voice session, sees the orb listening, freezes. No
     conversation starter suggestions visible. Awkward silence.
     Fix: Add 3-4 tappable chips ("What's nearby?", "Find me food",
     "Tell me about this area"). 1 hour.

  I. Proactive narration interrupts a natural moment
     User is mid-question, AI finishes responding, narration immediately
     fires about a nearby temple. User feels interrupted.

  J. Navigation bottom card doesn't show ETA
     Investor asks "how far is it?" — the card only shows step distance,
     not total remaining distance or time estimate.
